\section{Related Work and Gap Analysis}
\label{sec:related}

Agent identity sits at the intersection of decentralized identity, capability-based authorization, and service mesh security. This section surveys seven categories of prior work, identifies the specific property each fails to provide, and synthesizes the gap that \aip{} addresses. Table~\ref{tab:comparison} summarizes the analysis across seven dimensions.

\subsection{W3C Decentralized Identifiers}
\label{sec:did}

W3C Decentralized Identifiers (DIDs)~\citep{w3c2026did} provide self-sovereign, blockchain-anchored identity without reliance on centralized registries. DID v1.1 reached Candidate Recommendation in March 2026, yet adoption remains disappointing. Among early adopters, 34\% of users experienced lockout within six months due to lost or mismanaged cryptographic keys, and 62\% of non-technical users reported confusion with wallet management. Block's abandonment of its ``Web5'' DID initiative in late 2024 signaled that even well-funded efforts could not overcome these friction points.

For AI agents, the UX failures are irrelevant: agents are software, and key management is a trivial engineering problem rather than a human factors challenge. However, the structural problems persist. Most DID methods depend on blockchain infrastructure for resolution, introducing latency, cost, and availability concerns that are incompatible with high-throughput agent delegation. Trust bootstrapping remains circular: a DID is self-certifying, but verifiers need an external reason to trust it, which reintroduces the centralized trust anchors DIDs were designed to eliminate. Only 37\% of jurisdictions have clear regulatory frameworks for blockchain-based identity, creating compliance uncertainty for enterprise deployments.

\subsection{OAuth 2.0/2.1}
\label{sec:oauth}

OAuth 2.1~\citep{ietf2024oauth21} is the most widely deployed authorization framework in production systems. The \mcp{} specification adopted OAuth 2.1 with PKCE as its authorization mechanism in 2026~\citep{anthropic2024mcp}, and \atoa{} supports RFC 8693 token exchange~\citep{google2025a2a}. For human-to-service authentication within a single trust domain, OAuth remains the right choice.

Three structural limitations prevent OAuth from serving as the agent identity layer. First, every trust domain requires its own authorization server, and there is no mechanism for cross-domain token verification without pre-established federation agreements. In an open multi-agent ecosystem where agents from different organizations collaborate dynamically, this prerequisite cannot be met. Second, OAuth tokens are opaque to intermediaries: when Agent A delegates to Agent B via token exchange, the resulting token carries no record of the delegation chain, and B's downstream calls lose the context of A's original authorization. Third, OAuth provides no holder-side scope attenuation. Only the authorization server can narrow scopes at issuance time. A token holder cannot further restrict its own permissions before passing the token downstream, which is precisely what safe agent delegation requires.

\subsection{Macaroons}
\label{sec:macaroons}

Macaroons~\citep{birgisson2014macaroons} introduced holder attenuation as a first-class primitive: a token holder can append caveats that further restrict the token's authority before passing it downstream, without coordinating with the issuer. This insight is foundational. Google DeepMind's ``Intelligent AI Delegation'' paper~\citep{deepmind2026delegation} builds directly on macaroons, proposing Delegation Capability Tokens (DCTs) that encode attenuated authority chains for agent-to-agent delegation.

DeepMind's validation lends credibility to the attenuation-first design philosophy, but the underlying cryptographic model has three limitations that prevent production deployment. First, macaroons use HMAC-based verification, which requires the verifier to possess the root secret. This creates a single point of compromise: any verifier that can validate a token also holds the key material to forge one. Second, macaroon caveats are simple key-value predicates, insufficient for expressing complex delegation policies such as ``allow read access to financial data only during business hours and only for queries under \$10,000.'' Third, while DeepMind's DCT proposal is conceptually sound, it remains a position paper with no shipped protocol, no token format specification, and no reference implementation.

\subsection{UCAN}
\label{sec:ucan}

User Controlled Authorization Networks (UCANs)~\citep{ucan2024spec} implement decentralized delegation via nested JWTs with DID-based identity. Each delegation step wraps the parent JWT inside a new one, creating a verifiable chain of authority reductions. The philosophy is sound: users (not servers) control who gets what authority, and delegation is cryptographically traceable without centralized coordination.

Four limitations prevent UCANs from serving as the agent identity layer. First, DID dependency inherits the full complexity of DID resolution (Section~\ref{sec:did}), including blockchain infrastructure requirements and circular trust bootstrapping. Second, nested JWT encoding creates token bloat that grows quadratically with delegation depth: a five-hop chain requires five nested signatures, each wrapping the full payload of the previous hop. Third, the UCAN ecosystem is JS-centric and tightly coupled to web3 storage platforms like Storacha, limiting adoption in enterprise environments running Python, Java, or Rust agent stacks. Fourth, UCAN capabilities are expressed as flat URIs (\texttt{msg/send}, \texttt{file/read}), which lack the expressiveness to encode conditional policies, budget constraints, or temporal restrictions that agent delegation demands.

\subsection{Biscuit}
\label{sec:biscuit}

Biscuit~\citep{biscuit2024spec} is an authorization token format from the Eclipse Foundation that combines Ed25519 public-key signatures, a Datalog policy language, and an append-only block structure. Each block in a Biscuit token is signed by a new ephemeral key, and any holder can append a block that further restricts (but never expands) the token's authority. The Datalog authorizer evaluates all blocks together against a set of policy rules, producing an allow/deny decision. Among existing solutions, Biscuit gets the most right: public-key verification eliminates the shared-secret problem of macaroons, Datalog provides expressive policy evaluation, and append-only blocks naturally model delegation chains.

Biscuit is a token format, not an identity protocol. It defines no identity resolution mechanism (how to discover an agent's public key from its name), no protocol bindings (how to transport tokens over \mcp{}, \atoa{}, or HTTP), and no provenance concept (how to bind the outcome of a delegated action back to the authorization chain). These gaps mean that two independent deployments using raw Biscuit tokens have no interoperability path. Section~\ref{sec:biscuit-delta} details precisely what \aip{} adds on top of Biscuit's cryptographic foundation.

\subsection{SPIFFE/SPIRE}
\label{sec:spiffe}

The Secure Production Identity Framework for Everyone (SPIFFE)~\citep{spiffe2024} provides workload identity in cloud-native environments. SPIRE (the SPIFFE Runtime Environment) issues X.509 or JWT SVIDs (SPIFFE Verifiable Identity Documents) to workloads based on attestation of their runtime environment. Production deployments at Uber, Stripe, and Netflix demonstrate that SPIFFE scales to tens of thousands of workloads with sub-second identity verification.

Three factors limit SPIFFE's applicability to agent identity. First, SPIRE requires dedicated infrastructure (attestation nodes, registration servers, certificate authorities) that creates a significant deployment burden for organizations that simply want to add identity to their agent workflows. Second, X.509 certificate issuance introduces latency that is incompatible with ephemeral agent creation: an orchestrator that spawns a specialist agent for a single tool call cannot wait for certificate signing and distribution. Third, SPIFFE addresses workload identity within infrastructure boundaries but provides no mechanism for cross-protocol identity flow. A SPIFFE SVID issued to an agent calling an \mcp{} tool server has no defined mapping to the \atoa{} agent card that the same agent presents when collaborating with peers.

\subsection{IETF Drafts}
\label{sec:ietf}

Four IETF Internet-Drafts published in early 2026 address agent identity, reflecting growing recognition of the problem within the standards community. The AIMS draft~\citep{ietf2026aims} (draft-klrc-aiagent-auth) proposes a conceptual model composing WIMSE, SPIFFE, and OAuth into a unified agent authentication architecture, but its authorization section reads ``TODO Security,'' leaving the hardest problem unspecified. The WIMSE AI Agent Identity draft~\citep{ietf2026wimse} introduces a Dual-Identity Credential that binds an agent's identity to its owner's identity, solving the ``who controls this agent'' problem for two-party interactions, but provides no mechanism for multi-hop delegation where Agent A authorizes Agent B, which then delegates to Agent C.

The Agentic JWT draft~\citep{ietf2026agenticjwt} extends standard JWTs with agent-specific claims (agent identifier, owner binding, task context), but JWTs are immutable after signing. A delegatee cannot attenuate its received authority before passing it downstream without minting an entirely new token, which breaks the cryptographic chain linking the delegatee back to the original authorization. SCIM for agents~\citep{ietf2026scim} addresses provisioning lifecycle (creation, suspension, deprovisioning) rather than runtime authorization, making it complementary to rather than competitive with \aip{}. Collectively, these drafts define fragments of the solution space but leave the core problem unsolved: no draft provides a concrete token format with holder-attenuable delegation, cross-protocol bindings, and provenance tracking.

\subsection{Gap Summary}
\label{sec:gap}

The preceding review reveals that agent identity requires seven properties that no existing solution fully provides:

\begin{enumerate}[leftmargin=*,nosep]
  \item \textbf{Public-key verification without shared secrets.} Verifiers must validate tokens without possessing root key material. Macaroons and OAuth fail this requirement.
  \item \textbf{Holder-side scope attenuation.} A token holder must be able to narrow its own authority before delegating, without contacting the issuer. OAuth and standard JWTs fail this requirement.
  \item \textbf{Expressive policy language.} Delegation constraints must express conditions beyond flat scopes or key-value pairs (temporal bounds, budget limits, tool allowlists with parameters). Macaroons and UCAN fail this requirement.
  \item \textbf{Cross-protocol identity flow.} A single identity must be verifiable across \mcp{} tool calls, \atoa{} agent collaboration, and HTTP API requests. Every surveyed solution addresses at most one transport.
  \item \textbf{Provenance binding.} The outcome of a delegated action must be cryptographically bound to the authorization chain that permitted it. No surveyed solution provides this.
  \item \textbf{No blockchain or heavy infrastructure dependency.} The solution must work without blockchain networks or dedicated infrastructure clusters. This rules out most DID methods and SPIFFE/SPIRE for lightweight deployments.
  \item \textbf{Agent lifecycle awareness.} Tokens must accommodate ephemeral agents with short-lived grants and bounded delegation depth, not just long-lived service accounts. OAuth and SPIFFE assume persistent identities.
\end{enumerate}

\subsection{Relationship to Biscuit}
\label{sec:biscuit-delta}

\aip{} uses Biscuit~\citep{biscuit2024spec} as the cryptographic primitive for its chained token mode, in the same way that SSH uses Ed25519 as its signature algorithm. Biscuit provides the append-only block structure, Ed25519 signature chaining, and Datalog policy evaluation that form \aip{}'s verification core. What \aip{} adds is everything required to turn a token format into an identity protocol: (1) an identity resolution scheme (\texttt{aip:web} for DNS-anchored discovery, \texttt{aip:key} for self-certifying ephemeral agents), (2) protocol bindings that define how tokens flow over \mcp{}, \atoa{}, and HTTP with standardized headers and error codes, (3) completion blocks that bind the outcome of a delegated action back to the authorization chain, creating a verifiable provenance record, (4) a policy profile system (Simple, Standard, Advanced) that makes Datalog accessible to non-expert users while preserving full expressiveness for advanced deployments, and (5) a compact JWT mode for single-hop cases where the overhead of Biscuit's block structure is unnecessary. This relationship means \aip{} inherits Biscuit's formal security properties (proven attenuation guarantees, bounded Datalog evaluation) while addressing the protocol-level gaps that prevent raw Biscuit tokens from interoperating across agent ecosystems.

\section{The Agent Identity Protocol}
\label{sec:protocol}

\subsection{Identity Scheme}
\label{sec:identity}

\aip{} defines two identity resolution methods that share a common token format and verification path.

\paragraph{DNS-based identities} use the scheme \texttt{aip:web:<domain>/<path>} and resolve via HTTPS to a well-known endpoint. The identifier \texttt{aip:web:jamjet.dev/agents/research-analyst} resolves to \texttt{https://jamjet.dev/.well-known/aip/agents/research-analyst.json}. DNS-based identities are appropriate for long-lived agents operated by organizations that control a domain name.

\paragraph{Self-certifying identities} use the scheme \texttt{aip:key:ed25519:<multibase>}, where the identifier is the public key itself. No resolution step is required: the verifier extracts the key directly from the identifier. Self-certifying identities are appropriate for ephemeral sub-agents spawned by an orchestrator for a single task (Section~\ref{sec:delegation}).

Both resolution methods produce an \aip{} identity document: a JSON object containing the agent's public keys, delegation parameters, supported protocol bindings, and an expiration timestamp. Figure~\ref{fig:identity-doc} shows an abbreviated example.

Identity documents MUST include a \texttt{document\_signature} field: an Ed25519 signature computed over the RFC~8785~\citep{rfc8785} canonical form of the document with the \texttt{document\_signature} field excluded from the input. This self-signature provides content integrity independent of transport security. Even if the hosting domain is compromised (DNS hijack, CDN poisoning), an attacker cannot forge a valid identity document without the corresponding private key. Verifiers MUST check the document signature before trusting any fields.

\paragraph{Version compatibility.} Implementations parse the \texttt{aip} field as a semver \texttt{major.minor} string. Documents with a higher major version than the implementation supports MUST be rejected. Unknown fields MUST be ignored, enabling forward-compatible extension through minor version increments.

\subsection{Invocation-Bound Capability Tokens}
\label{sec:ibct}

The core contribution of \aip{} is the \ibct{}: a single token that fuses identity, attenuated authorization, and provenance into an append-only chain. An \ibct{} evolves through three block types as delegation proceeds.

\paragraph{Block 0 (Authority).} The root block is signed by the human or system that initiates the delegation chain. It declares the root identity, initial capability scopes, a budget ceiling, the maximum delegation depth, and an expiry timestamp. This block establishes the ceiling of authority for the entire chain.

\paragraph{Block N (Delegation).} Each intermediary agent appends a delegation block, signed with its own key. A delegation block names the delegator and delegatee, attenuates scope (narrowing capabilities from those granted by the parent block), and includes a mandatory \texttt{context} field describing the purpose of the delegation. Scope attenuation is cryptographically enforced: a delegation block that attempts to widen any capability beyond its parent block fails verification.

\paragraph{Block N+1 (Completion).} When work finishes, the executing agent may append a completion block recording the result hash, verification status, resource consumption, and cost. Completion blocks are optional; their trust semantics are detailed in Section~\ref{sec:completion}.

A completed \ibct{} answers five questions from a single artifact: Who authorized this action? Through which agents did the delegation flow? What constraints applied at each hop? What was the outcome? Was the outcome independently verified?

\subsubsection{Wire Formats}

\aip{} defines two wire formats that serve different deployment scenarios.

\paragraph{Compact mode} uses a standard JWT with \texttt{alg: EdDSA} and \texttt{typ: aip+jwt}. The payload carries \aip{}-specific claims: \texttt{iss} (issuer identity), \texttt{sub} (subject identity), \texttt{scope} (capability array), \texttt{budget\_usd} (spending ceiling), \texttt{max\_depth} (set to 0 to prohibit further delegation), and standard \texttt{iat}/\texttt{exp} timestamps. Compact mode covers single-hop cases: one agent calling one \mcp{} tool server, or a human granting authority to a single agent. Any JWT library that supports EdDSA can create and verify compact tokens.

\paragraph{Chained mode} uses the Biscuit token format~\citep{biscuit2024spec}. Each block is signed with an ephemeral Ed25519 key, and the append-only structure ensures that no party can remove or modify earlier blocks. Datalog policies in each block are evaluated together by the authorizer to produce an allow or deny decision. Chained mode is required for multi-hop delegation, cross-organization workflows, and any scenario that requires an audit trail of the full delegation chain.

\paragraph{Mode detection and upgrade.} Receivers detect the wire format from the token header: \texttt{typ: aip+jwt} indicates compact mode; Biscuit magic bytes indicate chained mode. Upgrading from compact to chained mode requires re-issuance: the issuer creates a new chained token (Block~0) using the same key, mapping compact JWT claims to Biscuit authority facts according to Table~\ref{tab:claim-map}. The original compact token is not embedded; this is a new token issuance with equivalent authority.

\subsection{Completion Blocks and Trust Model}
\label{sec:completion}

Completion blocks are signed by the executing agent, which is the party whose work is being reported. This means completion blocks are \textbf{self-reported claims}, not independently verified proofs. Two properties hold by construction: the block is \textit{tamper-evident} (any modification after signing is detectable) and \textit{attribution-bound} (the signature proves which agent made the claim, so a fraudulent report is attributable). However, the executing agent could misrepresent the result hash or cost without detection by the cryptographic layer alone.

\aip{} defines three trust escalation levels for completion blocks:

\begin{enumerate}[leftmargin=*,nosep]
  \item \textbf{Self-reported (default).} The executing agent reports its own results. Sufficient for internal, trusted environments where agents operate under a single organization's control.
  \item \textbf{Counter-signed.} The delegator (parent agent) independently verifies the result and appends a counter-signature block attesting to correctness. This adds one block to the chain containing the delegator's identity and a verification flag.
  \item \textbf{Third-party attested.} An external verifier (an \ldp{} peer verification node~\citep{prakash2025ldp}, a human reviewer, or an audit service) signs an attestation block. The \texttt{verification\_status} field maps directly to \ldp{}'s verification status enum, enabling seamless provenance bridging between the two protocols.
\end{enumerate}

Consumers of completion blocks SHOULD inspect the \texttt{verification\_status} field and weight trust accordingly. A \texttt{peer\_verified} or \texttt{tool\_verified} completion block carries more evidential weight than an unverified self-report.

\subsection{Budget Semantics}
\label{sec:budget}

Budget values in \ibct{}s are expressed as integer cents. This choice is forced by the Biscuit Datalog engine, which has no floating-point type. A budget of \$0.50 is encoded as \texttt{budget(50)}.

Budget fields represent \textbf{per-token authorization ceilings}, not running balances. When Agent~A delegates to Agent~B with \texttt{budget:50}, Agent~A asserts that B is authorized to spend up to 50 cents on this specific task. Agent~A is responsible for partitioning its own budget ceiling across sub-delegations. At invocation time, the verifier (\mcp{} server, \atoa{} receiver) checks that the declared budget in the presented token is non-negative; it does not track cumulative spend across invocations. At completion time, the completion block records actual \texttt{cost\_usd} for audit, not enforcement.

Aggregate spend enforcement is the responsibility of the runtime (\eg{}, an orchestration platform's existing cost-tracking subsystem), not the token. The analogy is a credit card: the \ibct{} authorizes spending up to a ceiling, the merchant (verifier) checks the limit, but the bank (runtime) tracks the running balance.

\subsection{Policy Profiles}
\label{sec:policy}

Datalog policies in chained-mode tokens use one of three profiles, ordered by increasing expressiveness and complexity.

\paragraph{Simple profile.} Users specify parameter values (tool allowlist, budget ceiling, maximum depth, expiry) and the library generates canonical Datalog checks. The four canonical templates are normative; implementations MUST generate exactly these patterns to ensure cross-implementation interoperability:

\begin{lstlisting}[style=protocol]
check if tool($t), ["search","browse"].contains($t);
check if budget($b), $b <= 50;
check if depth($d), $d <= 3;
check if time($t), $t <= 2026-03-22T12:00:00Z;
\end{lstlisting}

No Datalog knowledge is required from the user. The Simple profile covers the majority of single-organization delegation scenarios.

\paragraph{Standard profile.} A curated subset of Datalog that permits conjunctive conditions and set membership tests but prohibits recursion and unbounded evaluation. Example policies include trust-domain restrictions (\texttt{check if delegator(\$d), trust\_domain(\$d, \$dom), ["internal"].contains(\$dom)}) and conditional tool access based on delegator attributes.

\paragraph{Advanced profile.} Full Biscuit Datalog with an evaluation depth limit of 1,000 iterations. This profile supports recursive policy rules, cross-block fact derivation, and arbitrary constraint combinations. It is opt-in and intended for enterprise deployments with complex governance requirements. Verifiers that do not support the Advanced profile MUST reject tokens containing non-Standard Datalog rather than silently ignoring policy blocks.

\subsection{Protocol Bindings}
\label{sec:bindings}

\aip{} defines concrete bindings for three transport protocols, ensuring that a single identity and token format works across the agent ecosystem.

\paragraph{\mcp{} binding.} Clients include the \ibct{} in the \texttt{X-AIP-Token} HTTP header on every tool call. The server performs five verification steps: (1)~extract the token from the header (or fetch via \texttt{X-AIP-Token-Ref} for token-by-reference), (2)~resolve the issuer's identity document and verify all signatures, (3)~evaluate Datalog policies against the requested tool, (4)~for chained tokens, verify delegation depth, budget ceiling, and expiry at each block, and (5)~inject the verified identity into the request context for use by the tool implementation. \mcp{} servers declare \texttt{require\_aip: true} in their identity document to reject anonymous calls. Nine structured error codes distinguish authentication failures (HTTP~401: \texttt{aip\_token\_missing}, \texttt{aip\_token\_malformed}, \texttt{aip\_signature\_invalid}, \texttt{aip\_identity\_unresolvable}, \texttt{aip\_token\_expired}, \texttt{aip\_key\_revoked}) from authorization failures (HTTP~403: \texttt{aip\_scope\_insufficient}, \texttt{aip\_budget\_exceeded}, \texttt{aip\_depth\_exceeded}).

\paragraph{\atoa{} binding.} Agents advertise their \aip{} identity through the \texttt{aip\_identity} field in the \atoa{} agent card~\citep{google2025a2a}, which includes both the \aip{} identifier and a URL to the identity document. When submitting a task, the caller places the \ibct{} (with a delegation block appended for the receiving agent) in the \texttt{aip\_token} field of the task metadata. The receiver verifies the full chain, from root authority through all intermediate delegations to the scope granted for its task, and may further delegate if the remaining depth permits.

\paragraph{HTTP binding.} For generic HTTP APIs, the token is carried in the \texttt{Authorization} header using the \texttt{AIP} scheme: \texttt{Authorization: AIP <token>}. APIs that do not understand the \texttt{AIP} scheme ignore the header. For tokens exceeding 4\,KB (deep chains with complex policies), implementations MAY use token-by-reference via the \texttt{X-AIP-Token-Ref} header, which contains an HTTPS URL from which the receiver fetches the full self-authenticating token chain.

\subsection{Delegation Rules}
\label{sec:delegation}

Six rules govern how \ibct{}s propagate through delegation chains.

\textbf{Scope attenuation only.} Each delegation block MUST be a subset of its parent's capabilities. A block that attempts to widen any scope, increase the budget, or extend the expiry fails cryptographic verification. This is enforced by the Biscuit authorizer, which evaluates all blocks together and rejects any fact that contradicts an earlier attenuation.

\textbf{Bounded depth.} Block~0 declares a \texttt{max\_depth} value (default:~3). Delegation beyond this depth is rejected. The depth counter represents the maximum number of delegation blocks permitted, not the number of hops taken so far.

\textbf{Non-empty context.} Every delegation block MUST include a non-empty \texttt{context} field describing why the delegation is occurring. Verifiers MUST reject tokens with missing or empty context fields. This requirement exists for audit trail integrity: a completed \ibct{} should explain the purpose of each hop.

\textbf{Ephemeral grants for sub-agents.} When an orchestrator spawns a short-lived sub-agent, it generates an Ed25519 keypair, assigns the sub-agent an \texttt{aip:key} identity, and appends a delegation block with narrowed scope and a short TTL (typically minutes). The sub-agent uses the token for its work; the token auto-expires. No DNS registration is needed.

\textbf{Key rotation.} DNS-based identities support zero-downtime rotation through overlapping validity windows. The identity document lists multiple keys with \texttt{valid\_from} and \texttt{valid\_until} timestamps; tokens signed with any currently-valid key are accepted. Self-certifying identities do not support rotation: the key is the identity, so rotation means creating a new identity.

\textbf{Revocation and mutual authentication.} \aip{} v1 prefers short-lived tokens (under one hour) over revocation infrastructure, following the lesson from X.509 that revocation checking is the hardest part to get right and the first part to fail. For high-security scenarios, identity documents MAY declare a CRL endpoint. Mutual authentication in v1 uses DNS-based TLS verification: the caller resolves the receiver's identity document before sending and verifies that the TLS certificate matches the domain. Mutual authentication for self-certifying identities requires a challenge-response sub-protocol that is deferred to v2.
